## What This Project Does

**bp-cap-fiori** is a full-stack SAP Business Technology Platform (BTP) application that manages Business Partners. It allows users to create, view, update, block, and unblock business partner records (persons, organisations, and groups) through a Fiori Elements UI backed by a CAP OData V4 service running on Cloud Foundry.

The application replicates a simplified version of the SAP S/4HANA Business Partner master data concept — with addresses, roles, and tax numbers as child entities — and enforces role-based access control through XSUAA on BTP.

## Toolchain / Node Version

CDS 9's dependency chain (specifically the native `better-sqlite3` module pulled in by `@cap-js/sqlite@2.x`) needs a current Node LTS — an older Node has no prebuilt binary for it and the source-compile fallback fails (`node-gyp` needs Python's `distutils`, removed in Python 3.12+).

Managed via `nvm`, scoped to this project only (`.nvmrc` → `lts/*`) — it does **not** touch any system-wide/global Node install:

```bash
nvm install   # reads .nvmrc
nvm use       # switches this shell to the pinned version
```

Run `nvm use` (or let a `.nvmrc`-aware shell hook do it automatically) before `npm install` or `npm run dev` in a fresh terminal.

## Local Dev Database

Dev uses file-based SQLite at `db/dev.sqlite` (switched from `:memory:` so changes made through the UI — e.g. Block/Unblock — survive a `cds watch` restart, which happens automatically on every file save).

`cds watch` does **not** auto-deploy/seed a file-based SQLite DB the way it does for `:memory:` — it only connects to whatever is already there. To bridge that gap, `package.json` has:

```json
"predev": "test -f db/dev.sqlite || npx cds deploy",
"dev":    "cds watch"
```

npm runs `predev` automatically before `dev`. It deploys the schema and loads all `db/data/*.csv` seed files **only if `db/dev.sqlite` doesn't exist yet** — so a fresh clone works out of the box, and existing local data is never wiped by a normal restart.

To reset to pristine seed data:
```bash
rm -f db/dev.sqlite*
npm run dev   # predev detects the missing file and reseeds from db/data/*.csv
```

`db/dev.sqlite*` is gitignored — never commit it.

**After a major `@sap/cds` version bump**, delete and redeploy the dev DB even if it already exists. CAP's own draft/system tables can gain columns between major versions (e.g. CDS 9 added `DraftMessages` to `DraftAdministrativeData`) — an old DB file will fail with `no such column` errors on draft operations until it's redeployed against the current model: `rm -f db/dev.sqlite* && npx cds deploy`.

## Architecture

### Runtime Stack

| Layer | Technology |
|---|---|
| UI | SAP Fiori Elements (List Report + Object Page, OData V4) |
| Routing | SAP Approuter (`@sap/approuter`) |
| Service | SAP CAP Node.js (`@sap/cds` ^9), Express 5 |
| AI Access | MCP server (`@gavdi/cap-mcp`) at `/mcp` — see **MCP Server** below |
| Auth (dev) | CAP mocked auth (HTTP Basic) |
| Auth (prod) | XSUAA (`@sap/xssec` ^4 + `passport`) |
| DB (dev) | SQLite, file-based at `db/dev.sqlite` (`@cap-js/sqlite` ^2) — persists across restarts |
| DB (prod) | SAP HANA Cloud (HDI container, `hdb` / `@sap/hana-client`) |
| Hosting | SAP BTP Cloud Foundry (EU10 region) |

### Request Flow

```
Browser
  → Approuter  (app/bp-manager/xs-app.json)
      → /api/v1/*  forwarded to CAP srv  (destination: bp-cap-srv, forwardAuthToken: true)
          → srv/bp-service.js  (BEFORE / AFTER / ON hooks)
              → SQLite file (db/dev.sqlite)  (dev)
              → SAP HANA HDI container  (prod)
```

The approuter enforces XSUAA authentication at route level and handles the OAuth2 token exchange before forwarding requests to the CAP service.

## Data Model (`db/schema.cds`)

Namespace: `com.bp.manager`

### Entities

| Entity | Key | Description |
|---|---|---|
| `BusinessPartners` | `cuid` (UUID) | Core entity. Has `category`, `status`, `isBlocked`, name fields, and compositions to addresses/roles/tax numbers |
| `BPAddresses` | `cuid` | Postal addresses (country, city, postal code, street, house number) |
| `BPRoles` | `cuid` | Business partner roles with validity dates (e.g. customer FLCU01, vendor FLVN01) |
| `BPTaxNumbers` | `cuid` | Tax identification numbers per tax type (e.g. DE0 for German VAT) |
| `BPCategories` | `code String(1)` | Code list: `1`=Person, `2`=Organisation, `3`=Group |
| `BPStatuses` | `code String(10)` | Code list: `ACTIVE`, `BLOCKED` |

### Key Field Behaviours

- **`fullName`** — stored/computed field (not a DB function). Written by the handler on every CREATE and UPDATE when name-related fields change. For persons: `firstName + lastName`. For orgs/groups: `organizationName`.
- **`statusCriticality`** — `virtual Integer` field, never persisted. Set in the `AFTER READ` hook to drive Fiori traffic-light colouring: `1`=red (blocked), `3`=green (active), `0`=neutral.
- All entities use `cuid` (UUID primary key) — required for CAP draft support and Fiori Elements navigation.
- `BusinessPartners` also uses `managed` mixin, adding `createdAt`, `createdBy`, `modifiedAt`, `modifiedBy` audit fields (`createdBy`/`modifiedBy` are excluded from the service projection).

## Service Layer (`srv/bp-service.cds`)

Exposed at path `/api/v1/bp`. Requires `authenticated-user` at minimum.

### Projection Changes vs Schema

`BusinessPartners` in the service flattens association keys:
- `category.code as categoryCode` and `category.name as categoryName`
- `status.code as statusCode` and `status.name as statusName`

This avoids the need for `$expand` in Fiori value help requests.

Draft is enabled only on `BusinessPartners` (`@odata.draft.enabled`). Child entities (`BPAddresses`, `BPRoles`, `BPTaxNumbers`) participate in draft automatically via composition.

`blockBP`/`unblockBP` are declared **inside** the entity's `actions { }` block, which compiles them as OData V4 *bound* actions (`IsBound="true"`, no `ActionImport`). They must stay bound — see **Bound Actions & Drafts Gotcha** below for what breaks if they're moved back to the service level. Each carries `@Common.SideEffects: { TargetEntities: ['_it'] }` so a Fiori client refetches the whole BP (including the virtual `statusCriticality`) after a successful call, instead of leaving stale cached values on screen.

### Authorization per Operation

| Role | READ | WRITE / CREATE | `blockBP` | `unblockBP` |
|---|---|---|---|---|
| `BP_VIEWER` | Yes | No | No | No |
| `BP_PROCESSOR` | Yes | Yes | Yes | No |
| `BP_ADMIN` | Yes | Yes | Yes | Yes |

## Business Logic (`srv/bp-service.js`)

Extends `cds.ApplicationService`. All hooks registered inside `init()`.

| Hook | Entity | What it does |
|---|---|---|
| `BEFORE CREATE` | BusinessPartners | Validates category is provided; validates `organizationName` (cat 2/3) or `lastName` (cat 1); computes `fullName`; defaults `status_code='ACTIVE'` and `correspondenceLang='EN'` |
| `BEFORE UPDATE` | BusinessPartners | Only triggers if name fields are in the PATCH; re-fetches current row from DB to merge missing fields; recomputes `fullName` |
| `AFTER READ` | BusinessPartners | Sets `statusCriticality`: `1` if `isBlocked`, `3` if `status_code='ACTIVE'`, `0` otherwise |
| `BEFORE READ` | BusinessPartners | Caps `$top` at 500 rows to prevent unbounded queries |
| `ON blockBP` / `ON unblockBP` | BusinessPartners **and** BusinessPartners.drafts (bound action) | Shared `blockHandler(toBlocked)` factory. Guards against a no-op (already blocked/active). Always updates the persisted **active** row. Also mirrors `isBlocked`, `status_code`, `statusCode`, `statusName` into the drafts table if a draft is open — see below. `unblockBP` restricted to `BP_ADMIN`. |
| `ON getBPSummary` | — (unbound function) | Returns aggregate counts: `totalBPs`, `activeBPs`, `blockedBPs`, `persons`, `orgs` |

### Bound Actions & Drafts Gotcha

Registering `blockBP`/`unblockBP` only against the active `BusinessPartners` entity is not enough — invoking them while a record is being edited (draft open, `IsActiveEntity=false`) routes to `BusinessPartners.drafts` instead, and CAP returns `501 no handler for "blockBP BPService.BusinessPartners.drafts"` if nothing is registered there. Both handlers are registered for `[BusinessPartners, BusinessPartners.drafts]`.

The drafts shadow table also **materializes** `categoryCode`/`categoryName`/`statusCode`/`statusName` as flat physical columns — unlike the active entity, where they're a live SQL view via the `category`/`status` associations. Patching only `status_code` on the drafts row leaves `statusCode`/`statusName` stale (the edit-mode form would keep showing the old status text even though `isBlocked` updated correctly). The handler looks up the correct name from `BPStatuses` and patches all four fields on the draft.

## Fiori Elements UI (`srv/annotations.cds`)

All OData annotations that drive the Fiori Elements UI are in this single file. Nothing UI-related lives in `schema.cds` or `bp-service.cds`.

### List Report Page
- **Table columns**: BP Number, Name (fullName), Category, Status (with traffic-light icon via `statusCriticality`), Search Term, Blocked, Created On
- **Filter bar fields**: BP Number, Full Name, Category, Status, Blocked, Search Term

### Object Page
Five facets/tabs:

| Facet | Content |
|---|---|
| General Information | BP Number, Category, Status, Blocked flag, Search Term, Correspondence Language |
| Name Details | Organisation Name, First Name, Last Name, Full Name |
| Addresses | Sub-table of `BPAddresses` |
| Roles | Sub-table of `BPRoles` |
| Tax Numbers | Sub-table of `BPTaxNumbers` |

Header actions (shown as buttons): **Block BP**, **Unblock BP** — available in both display and edit mode.

Referenced in `UI.Identification` by their plain bound-action qualified name, `BPService.blockBP` / `BPService.unblockBP`. This must match how the action is declared in `bp-service.cds`: bound actions are referenced by qualified name; the `<Service>.EntityContainer/<Entity>_<action>` path form is only valid for *unbound* actions exposed as an `ActionImport`. Using the wrong form doesn't error at compile time — it fails silently in the browser (UI5 throws `Unknown action import` and never issues the OData request), so if a button appears clickable but no `$batch` call fires, check this first.

### F4 Value Helps
- `categoryCode` → `BPCategories` entity (code + name displayed)
- `statusCode` → `BPStatuses` entity (code + name displayed)

Both use `Common.ValueListWithFixedValues: true` so the input renders as a dropdown.

## MCP Server (`srv/mcp.cds`)

Exposes `BPService` to MCP-compatible AI agents (e.g. Claude Desktop) via the [`@gavdi/cap-mcp`](https://github.com/gavdilabs/cap-mcp-plugin) plugin. Endpoint: `http://localhost:4004/mcp` (health at `/mcp/health`).

Configured in `package.json` under `cds.mcp`:
- `name`: `bp-manager-mcp`
- `auth: "inherit"` — MCP requests authenticate exactly like OData requests (HTTP Basic in dev, XSUAA/JWT in prod) and inherit the same `@restrict` role checks. A tool that a role has no grant for isn't just rejected — it's filtered out of `tools/list` entirely for that user, so it never shows up as available in the first place.

`@mcp` annotations live in their own file, `srv/mcp.cds`, mirroring how `annotations.cds` isolates `@UI` concerns — never add `@mcp` annotations to `bp-service.cds`.

What's exposed:

| Artifact | MCP form | Notes |
|---|---|---|
| `BusinessPartners` | Resource template (`filter`,`orderby`,`select`,`top`,`skip`,`expand`) + `query`/`get` wrapper tools | Deliberately **not** wrapped for `create`/`update` — it's draft-enabled with a multi-step Fiori edit flow; creates/updates should go through the UI, not a raw tool call |
| `BPCategories` / `BPStatuses` | Static resources (`resource: []`) | Small code lists, no query params needed |
| `blockBP` / `unblockBP` | Tools (`block-business-partner` / `unblock-business-partner`) | `elicit: ['confirm']` — the plugin requires the calling MCP client to support confirmation elicitation and refuses to execute otherwise (`Client does not support form elicitation`), rather than silently skipping the confirmation step |
| `getBPSummary` | Tool (`get-bp-summary`) | Read-only, no elicitation |

Field-level `@mcp.hint` annotations on `BusinessPartners` explain the coded values (`categoryCode` 1/2/3, `statusCode` ACTIVE/BLOCKED) an LLM agent can't infer from the field name alone, and steer agents toward the block/unblock tools instead of writing `isBlocked` directly.

### Bound-Action Annotation Syntax Gotcha

Annotating a bound action (declared inside an entity's `actions { }` block) from an external file needs `annotate <Entity> with actions { <action> @(...); };` — **not** `annotate <Entity>.actions { ... }`, which silently fails: it compiles without error but produces an `ext-undefined-def` warning and the annotation never merges into the actual action definition. Verify with `npx cds compile srv --to json` and check the annotation appears on the actual `actions.<name>` object, not just as an orphaned entry under `"extensions"`.

### Testing with the Claude Code CLI

Register the running dev server as a local MCP server (HTTP transport + Basic Auth header, since dev auth is mocked and doesn't do OAuth):

```bash
claude mcp add --transport http bp-manager http://localhost:4004/mcp \
  --header "Authorization: Basic $(echo -n 'admin@test.com:x' | base64)"
```

Verify: `claude mcp get bp-manager` (should show `Status: ✔ Connected`). Swap the base64'd credentials to test as `viewer@test.com`/`processor@test.com` and confirm RBAC — a viewer's `tools/list` won't even include `block-business-partner`/`unblock-business-partner`.

**A newly-registered MCP server is not hot-loaded into an already-running Claude Code session** — tool discovery happens at session start. To test without disrupting an active session, spawn a fresh headless process instead:

```bash
claude -p "Using the bp-manager MCP server, get the Business Partner summary counts." \
  --allowedTools "mcp__bp-manager__get-bp-summary"
```

Headless (`-p`) runs need `--allowedTools` listing the exact `mcp__<server>__<tool>` names — otherwise it can't get permission approval non-interactively. This is also directly relevant to `elicit: ['confirm']`: headless mode has no way to show an interactive confirmation dialog, so calling `block-business-partner`/`unblock-business-partner` via `-p` **correctly declines and does not execute**, rather than silently auto-confirming. To actually exercise a write tool end-to-end, use an **interactive** `claude` session, where the confirmation prompt can be shown and approved.

If you see `this workspace has not been trusted`, either run `claude` interactively once in this directory and accept the trust dialog, or it'll just skip project-level `.claude/settings.json` permissions for that headless run (harmless for read-only testing, but `--allowedTools` still works regardless).

### Manual curl Testing

A bare `curl http://localhost:4004/mcp` (no credentials) returns `401 Unauthorized` with a `WWW-Authenticate: Bearer resource_metadata=...` header — this is `auth: "inherit"` working as intended, not a bug. `curl http://localhost:4004/mcp/health` deliberately skips auth and returns `{"status":"UP"}`, so it can be used as an unauthenticated liveness check without proving anything about the MCP protocol layer itself. `/mcp` is a JSON-RPC POST endpoint, not a browsable page — a `GET` with valid Basic Auth credentials still 404s; use the `claude mcp add`/`claude -p` flow above (or a JSON-RPC POST body) to actually exercise it.

## CDS 8 Import Gotcha

`sap.common` is a `context` in `@sap/cds/common`, not a top-level namespace. Destructure it explicitly:

```cds
// correct
using { cuid, managed, Country, sap.common.CodeList } from '@sap/cds/common';

// wrong — worked in CDS 7, breaks in CDS 8
using { cuid, managed, sap.common } from '@sap/cds/common';
```

After importing `sap.common.CodeList` this way, reference it as just `CodeList` (not `sap.common.CodeList`) in entity definitions.

## CDS 9 / Express 5 Upgrade Gotcha

CDS 9 requires Express 5 (needed to satisfy `@gavdi/cap-mcp`'s peer dependency). `@sap/cds-fiori@1.x` predates Express 5 support: it builds Fiori-preview mock routes with a bare `'*'` wildcard (`path.join(appPath, dataSourceUri, '*')`), which Express 4's `path-to-regexp` accepted but Express 5's rejects outright at server startup:

```
PathError [TypeError]: Missing parameter name at index 48: /bp-manager/webapp/localService/annotation.xml/*
```

Fixed by requiring `@sap/cds-fiori@^2.3.0`, which detects the Express major version and uses `*splat` instead of bare `*` when running on Express 5. If you ever see this error again after touching CAP-related deps, check `@sap/cds-fiori`'s version first — it's not something in this project's own code.

Also required: `@cap-js/sqlite` bumped 1→2 (the 1.x line depends on a `@cap-js/db-service` that peer-caps `@sap/cds` at `<9`), `@sap/xssec` 3→4, and `eslint` 8→10 (a peer of CDS 9's bundled `@eslint/js`). See **Toolchain / Node Version** above for the Node-side fallout of this chain.

## Key Design Decisions

- **`fullName` is stored, not computed in SQL** — avoids DB-specific computed column syntax and keeps the value readable in raw DB queries. The handler writes it on every CREATE/UPDATE.
- **Annotations are isolated in `annotations.cds`** — keeps the domain model and service definition clean. Never add `@UI` or `@Common` annotations to `schema.cds` or `bp-service.cds`.
- **Category/status are associations, not plain strings** — enables F4 value helps and Fiori display names without `$expand` by projecting `category.code as categoryCode` in the service.
- **`statusCriticality` is virtual** — the value is always computed fresh on read; it cannot be stale because it is never stored.
- **Pagination guard is always on** — the `BEFORE READ` hook silently caps `$top` at 500 to protect against accidental full-table scans in production HANA.
- **Dev DB is a persistent file, auto-provisioned only when missing** — `:memory:` reset on every `cds watch` restart (which fires on every file save), destroying any Block/Unblock or other test data mid-session. File-based `db/dev.sqlite` plus a `predev` guard (`test -f db/dev.sqlite || npx cds deploy`) gives persistence across restarts while still working out of the box on a fresh clone.
- **`blockBP`/`unblockBP` must be bound actions, registered for both the active entity and `.drafts`** — declaring them at the service level compiles to unbound `ActionImport`s that Fiori Elements object-page buttons can't target a specific record with; registering the handler only against the active entity leaves the action broken (`501`) whenever it's invoked while a draft is open.
- **MCP annotations are isolated in `mcp.cds`, same reasoning as `annotations.cds`** — keeps `bp-service.cds` free of a third annotation domain and makes it obvious where to look for AI-facing exposure config.
- **MCP writes (`blockBP`/`unblockBP`) require client-side confirmation (`elicit: ['confirm']`)** — these mutate business state; requiring the calling agent to explicitly confirm before executing is a deliberate safety choice, not a default we should relax to make testing more convenient.
- **`nvm` + `.nvmrc` instead of upgrading the system-wide Node** — this project's toolchain now needs a newer Node than may be installed globally; scoping the upgrade via `nvm` avoids affecting other projects/tools on the same machine.
